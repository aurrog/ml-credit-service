# Credit Default ML Service

ML-сервис для прогнозирования дефолта по кредитным картам.


## Структура проекта

```text
├── app/
│   ├── __init__.py
│   └── model_handler.py
├── data/
│   └── UCI_Credit_Card.csv
├── logs/
│   └── api.log
├── models/
│   └── model.pkl
├── notebooks/
│   └── Untitled.ipynb
├── tests/
│   ├── __init__.py
│   └── test_api.py
├── main.py
├── Dockerfile
├── README.md
└── requirements.txt
```

## API
Сервис реализован на Flask.

## GET /health
Проверка работоспособности сервиса.

Пример запроса:
```commandline
curl.exe http://localhost:5000/health
```

Пример ответа:
```json
{
  "status": "ok"
}
```

## POST /predict
Принимает JSON с признаками клиента и возвращает предсказанный моделью класс и вероятность.

Пример запроса:
```commandline
curl.exe -X POST http://localhost:5000/predict -H "Content-Type: application/json" -d "{\"LIMIT_BAL\":20000,\"SEX\":2,\"EDUCATION\":2,\"MARRIAGE\":1,\"AGE\":24,\"PAY_0\":2,\"PAY_2\":2,\"PAY_3\":-1,\"PAY_4\":-1,\"PAY_5\":-2,\"PAY_6\":-2,\"BILL_AMT1\":3913,\"BILL_AMT2\":3102,\"BILL_AMT3\":689,\"BILL_AMT4\":0,\"BILL_AMT5\":0,\"BILL_AMT6\":0,\"PAY_AMT1\":0,\"PAY_AMT2\":689,\"PAY_AMT3\":0,\"PAY_AMT4\":0,\"PAY_AMT5\":0,\"PAY_AMT6\":0}"
```

Пример ответа:

```json
{
  "prediction": 0,
  "probability": 0.23,
  "model_version": "v1"
}
```

# Архитектура сервиса

В проекте используется монолитная архитектура.

Flask-приложение, загрузка модели и выполнение предсказаний находятся в одном сервисе. Такой подход выбран, т.к сложное масштабирование избыточно в учебном проекте, содержащем всего одну ML-модель и два эндпоинта.

Преимущества:

- простая структура
- простой запуск локально и в Docker
- достаточно для демонстрации ML в production

Микросервисная архитектура могла бы использоваться в дальнейшем, при развитии проекта. Имеет смысл разделить на отдельные сервисы инференс модели, логгирование, мониторинг, возможно также обучение/дообучение модели.

### RabbitMQ

RabbitMQ в текущей версии проекта не реализован, но может использоваться при масштабировании и повыщении нагрузки на сервис.

Возможные сценарии использования инструмента:
- асинхронная обработка батч-предсказаний
- постановка запросов на инференс в очередь
- снижение нагрузки на основной сервис

Например, API может принимать запрос пользователя, отправлять задачу в очередь RabbitMQ, а отдельный worker-сервис будет забирать задачу, выполнять предсказание и сохранять результат.

### Логирование и мониторинг

В проекте реализовано логгирование запросов и ответов API в json формате.

Логи сохраняются в logs/api.logs.
Пример лога:

```json
{
  "timestamp": "2026-05-05T15:01:50.233706",
  "endpoint": "/predict",
  "status": "success",
  "model_version": "v1",
  "prediction": 1,
  "probability": 0.5529592578612933}
```

В production-среде такие логи можно собирать и анализировать с помощью ELK-стека: Elasticsearch, Logstash и Kibana. Это позволит отслеживать количество запросов, ошибки API, распределение предсказаний и поведение модели во времени.

# DVC и MLflow

DVC может использоваться для контроля версий датасетов и моделей. Это позволяет воспроизводить обучение модели на прошлых версиях данных и откатывать модель, в случае если новая версия плохо показала себя в проде.

MLflow может использоваться для отслеживания экспериментов в обучении модели. Гиперпараметров, метрик и версий.

В данном проекте DVC и MLflow описаны концептуально, их реализация не требуется.

# A/B - тестирование

A/B-тестирование может использоваться для сравнения двух версий модели ML:

- контрольная группа - текущая модель (v1)
- тестовая группа - новая модель (v2)

В рамках данного проекта реализация A/B-тестирования не требуется, поэтому описывается только концепция.

### Разделение трафика

Пользовательские запросы можно случайно делить между двумя версиями модели в соотношении 50/50.

Тест можно проводить до момента, пока не накопится достаточного количества наблюдений для статистического анализа.

### Метрики

Для модели основная метрика это f1_score, т.к на нее ориентируемся и при обучении, и при настройке.

Дополнительная метрика precision важна, чтобы не выдавать плохие кредиты

Также можно учитывать бизнес-метрики, зависящие от конкретного направления применения сервиса, например:

1. Expected Loss (ожидаемые потери)
= сумма кредитов * вероятность дефолта

2. Approval Rate
доля одобренных заявок при фиксированном уровне риска

### Статистический анализ

Для сравнения результатов модели можно использовать t-test. 
H0: модели не имеют статистически значимых различий 
H1: у модели v2 метрики статистически значимо лучше, чем у v1

Для метрик необходимо построить доверительные интервалы. Если интервал для разницы метрик показывает устойчивое улучшение модели v2, то новая модель может считаться более успешной.

Критерий успеха:
p-value < 0.05 и улучшение F1
